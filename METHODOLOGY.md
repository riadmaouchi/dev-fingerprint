# Methodology — dev-fingerprint

## Scientific Objective

Estimate:

> **P(activity compatible with AI assistance | observed Git history)**

And **not**: `P(commit generated by AI)`.

The distinction is fundamental: we measure a process change,
not the provenance of the code.

---

## Pipeline Architecture

```
GitHub Commits
     │
     ▼
Per-commit metric extraction (StyleMetrics)
     │
     ├── Process signals (Level A/B) — direct commit metadata
     │   ├── files_changed
     │   ├── net_lines / total_churn
     │   ├── cross_module_ratio
     │   ├── is_refactor
     │   └── touches_tests / test_file_ratio
     │
     └── Style signals (Level C) — via AST/tree-sitter
         ├── comment_density, docstring_coverage
         ├── avg_identifier_length, error_handling_density
         └── style_score (CodeAnalyzer.copilot_score)
                    │
                    ▼
         Quarterly aggregation (BehaviorWindow)
                    │
                    ▼
         Change-point detection (PELT + CUSUM + EWMA + BOCPD)
         per signal independently — union of alarms
                    │
                    ▼
         Self-comparison test (Mann-Whitney U)
         recent author vs. historical author
                    │
                    ▼
         Fisher's method (Level A signals only)
                    │
                    ▼
         DriftResult — probabilistic interpretation
```

## Signal Hierarchy

### Level A — Process Signals (primary evidence)

Theoretical justification: these signals measure how code is produced,
not the appearance of the result. An AI assistant alters task granularity,
execution speed, and the topological consistency of changes.

| Signal | Description | Bypass resistance | Cross-language stability |
|--------|-------------|:-----------------:|:------------------------:|
| `median_files_per_commit` | Median number of files modified per commit | High | High |
| `large_commit_ratio` | Fraction of commits with \|net_lines\| > 200 | Moderate | High |
| `cross_module_ratio` | Normalized cross-module spread (0=same module, 1=max spread) | High | High |
| `refactor_ratio` | Fraction of refactoring commits (high churn, low net) | Moderate | High |
| `median_inter_commit_hours` | Median hours between consecutive commits | High | High |
| `commits_per_week` | Rolling commit frequency | High | High |

**Causal hypothesis**: an AI assistant lowers the cognitive cost of code generation,
resulting in larger and more cross-cutting commits. This is a plausible hypothesis,
not a demonstrated one.

**Confounding factors**:
- Role change (lead → contributor)
- Project phase (exploration → stabilization → maintenance)
- Adoption of a new framework
- Team size evolution

### Level B — Process Signals (useful but fragile)

| Signal | Description | Main weakness |
|--------|-------------|---------------|
| `test_touch_ratio` | Fraction of commits touching test files | Depends on project style and phase |
| `median_net_lines` | Median line delta per commit | High variance, depends on task type |

### Level C — Style Signals (baseline comparison only, not primary)

**These signals are kept for comparison with naive approaches.
They must NOT be used as primary evidence. See [CRITIQUE.md](CRITIQUE.md).**

| Signal | Description | Main weakness |
|--------|-------------|---------------|
| `style_score` | Composite CodeAnalyzer.copilot_score() | Indistinguishable from expert style |
| `comment_score` | Comment density | Function of project phase, not AI |
| `docstring_score` | Docstring coverage | Imposed by code review requirements |
| `verbosity_score` | Identifier length | Function of language conventions |
| `error_handling_score` | Error-handling density | Domain-dependent |
| `commit_style_score` | Fraction of conventional commits | Pre-AI standard (commitizen, etc.) |
| `function_style_score` | Inverse of function length | Varies by task type |

## Detection Methods

The pipeline uses **five distinct methods** in two roles:

| Role | Method | Type |
|------|--------|------|
| Change-point detection | PELT | Exact optimization |
| Change-point detection | CUSUM | Sequential control |
| Change-point detection | EWMA | Continuous early alert |
| Change-point detection | BOCPD | Online Bayesian inference |
| Global drift test | Mann-Whitney U + Fisher | Non-parametric |

The four change-point methods operate **signal by signal**, independently.
Their alarms are combined by union, then filtered by a relative magnitude
threshold (≥ 15%). The Mann-Whitney test operates on the recent-vs-historical
comparison and produces a combined p-value via Fisher's method (Level A signals only).

---

### Change-Point Method Comparison

| Criterion | PELT | CUSUM | EWMA | BOCPD |
|-----------|------|-------|------|-------|
| **Paradigm** | Global optimization | Sequential control | Continuous control | Bayesian inference |
| **Data assumption** | i.i.d. per segment (RBF) | Stationary Gaussian | Stationary Gaussian | Normal-Inverse-Gamma |
| **Detects abrupt shifts** | ✅ Optimal | ✅ Good | ✅ Good | ✅ Good |
| **Detects gradual drifts** | ⚠️ Weak | ✅ Strong | ⚠️ Weak | ❌ Weak |
| **Precise localization** | ✅ Exact | ⚠️ Delayed | ❌ Blurry | ✅ At jump point |
| **Interpretability** | Medium (optimization) | High (statistic) | High (statistic) | High (probability) |
| **Hyperparameter sensitivity** | `penalty` critical | `h` critical | `lambda_` moderate | `hazard` moderate |
| **Minimum series length** | 6 (2×min_size=3) | 4 | 4 | 4 |
| **Multiple change points** | ✅ Native | ⚠️ Reset on each alarm | ⚠️ Reset on alarms | ⚠️ Degrades after 1st |
| **Output** | Exact indices | Alarm indices | Out-of-control indices | Indices where P(r=0)>threshold |
| **Default parameters** | `penalty=3.0` | `k=0.5, h=5.0` | `λ=0.2, L=3σ` | `H=1/15, threshold=0.25` |

**Recommendation**: use PELT + CUSUM by default (complementary pair).
Add BOCPD for Level A signals when a directly interpretable change-point
probability is desired; EWMA for continuous monitoring without precise localization.

---

### PELT (Pruned Exact Linear Time)
- Exact optimization of segmentation cost over the entire series.
- `min_size=3`, `penalty=3.0` (BIC-like), relative magnitude threshold: 15%.
- **Strength**: optimal for a single dominant shift; reproducible results.
- **Weakness**: short series (< 12 points); does not detect slow drifts.

### CUSUM (Cumulative Sum Control Chart)
- Accumulates standardized deviations from the reference mean (first half).
- `k=0.5` (slack), `h=5.0` (threshold). Reset after each alarm.
- **Strength**: detects gradual drifts and small but persistent shifts.
- **Weakness**: detection lag; false positives if `h` is too low.

### EWMA (Exponentially Weighted Moving Average)
- Exponential moving average with steady-state UCL/LCL at 3σ.
- `lambda_=0.2`, `l_factor=3.0`. Initialized on first half.
- **Strength**: early alert; resistant to point outliers.
- **Weakness**: does not precisely localize the change point; less sensitive to large shifts.

### BOCPD (Bayesian Online Change Point Detection)
*Adams & MacKay, 2007 — "Bayesian Online Changepoint Detection"*

- Maintains a posterior distribution over the **current run length** r_t.
- Conjugate model: Normal-Inverse-Gamma (Student-t predictive).
- Alarm when P(r_t = 0 | x_{1:t}) > threshold (0.25 by default).
- `hazard = 1/15`: expected run length ≈ 4 years of quarterly data.
- **Strength**: directly interpretable probabilistic output; no frequentist calibration.
  Robust when segments have different variances.
- **Weakness**: global prior (whole-series mean) can mask a 2nd change point
  if the 1st has already reorganized the posterior. Very short series (< 8 pts)
  produce flat probabilities.

**Implementation note**: the prior is initialized on the mean and variance of the
entire series (diffuse prior). For series with `n < 8` windows, prefer PELT or CUSUM.

## Self-Comparison Test

### Protocol
- **Historical**: all windows except the `recent_n` last ones (default: 4).
- **Recent**: the `recent_n` last windows.
- **Test**: two-sided Mann-Whitney U (non-parametric, no normality assumption).
- **Minimum**: `min_historical=6` + `recent_n=4` = 10 windows minimum (~2.5 years).
- **Combination**: Fisher's method on p-values from Level A signals only.

### H₀ / H₁
- **H₀**: the recent distribution is drawn from the same distribution as the baseline.
- **H₁**: the recent distribution differs significantly from the baseline.

Rejecting H₀ implies a behavioral change. That change is **compatible** with AI
assistance, but also with a list of alternative factors.

### Alternative Factors to Systematically Consider

| Factor | Observable mechanism |
|--------|---------------------|
| Career stage | Increased delegation, fewer but larger commits |
| Role change | Lead → contributor, etc. |
| Project phase | Exploration vs. stabilization vs. maintenance |
| Team evolution | Size, culture, review process |
| Tool adoption | IDE, automated refactoring, CI/CD |
| Absence / return | Radically different patterns after a break |

## Linus Torvalds Case Study

Torvalds is a reference case study for **confounding factors**:

1. His role has evolved from active maintainer to "project guardian".
2. The Linux project has evolved massively: tooling, contributors, processes.
3. His publicly documented 2018 step-back creates a documented behavioral break
   independent of AI.

Any interpretation of a drift signal on Torvalds must account for these confounds
before considering the AI hypothesis. His window count (8 windows, all Q4 due to
Linux merge window clustering) also falls below the 10-window minimum required
for the Mann-Whitney / Fisher test — he cannot be statistically tested.

## Validation Protocol

### What is demonstrable today
- ✅ Detect a statistically significant behavioral change relative to a developer's own baseline.
- ✅ Identify which signals changed and in which direction.
- ✅ Approximately localize the date of the change.

### What is only plausible
- ⚠️ Associating the change with the emergence of AI tools (temporal correlation).
- ⚠️ Distinguishing AI assistance from other confounding factors.

### What is speculative
- ❌ Asserting that a developer uses AI from the signal alone.
- ❌ Quantifying intensity of use.
- ❌ Differentiating between AI tools.

### Data Required for Rigorous Validation
1. **Control group**: developers explicitly declaring they use no AI assistant.
2. **Positive group**: developers explicitly declaring they use an AI assistant.
3. **Calibration**: minimum 30 labeled samples (15/class), Platt scaling
   or isotonic regression (`validation/calibration.py`).
4. **Cross-validation**: LODO leave-one-developer-out (`validation/cross_validate.py`).

## Project Failure Risks

1. **Insufficient ground truth**: without verifiable declarations, any calibration
   is circular. This is the primary risk.

2. **Temporal fragility**: LLM patterns evolve. A 2023 calibrator will likely be
   invalid in 2027.

3. **Unobserved confounds**: GitHub contains no information about role changes
   or developers' personal lives.

4. **Developer adaptation**: prolonged AI use progressively reduces the signal
   (the workflow normalizes).

5. **Dominant individual variance**: for some developers, the variance within
   their own commits exceeds the AI/human difference.
