# Critique of Naive AI-Detection Approaches in Code

This document explains why intuitive approaches to detecting AI assistance in code
are scientifically indefensible. It provides the methodological justification for
the choices made in this project.

---

## Working Hypothesis

We are NOT attempting to detect whether a commit was generated by an AI.

We are estimating:

> **P(activity compatible with AI assistance | observed Git history)**

This distinction is fundamental. Code generated by an AI and code written by an
expert developer can be indistinguishable. The only reliable observable is the
**development process**, not the **code style**.

---

## 1. Detection by Code Style

### Description
Searching commit diffs for characteristics supposedly typical of AI-generated code:
function length, indentation, cyclomatic complexity, etc.

### Bias
- A senior Python developer with a strong PEP8 culture naturally produces short,
  well-structured functions — exactly the style attributed to LLMs.
- Code style depends on the language, framework, team conventions, developer
  seniority, and task type (feature vs. hotfix vs. refactoring).

### False Positives
- Any developer who follows modern ecosystem conventions.
- Any developer who has adopted a strict linter (black, gofmt, rustfmt).
- Any developer who has changed teams or projects with different conventions.

### False Negatives
- A Copilot user who systematically reworks AI suggestions to match their personal style.
- A Claude user who crafts prompts to produce "old-school" code.

### Trivial Bypass
A developer can disable or modify any AI suggestion before committing.
The final style in the diff is always the result of a human filter.

---

## 2. Detection by Comment Density

### Description
LLMs typically generate more comments and docstrings than organic developers.
Detect comment density above the historical average.

### Bias
- High-traffic open-source projects (Linux, CPython) have strict documentation
  conventions that predate LLMs.
- Comment density is a function of project phase (proof-of-concept vs. production)
  and change type (public API vs. internal).
- Code reviews and documentation requirements increase comments independently of AI.

### False Positives
- Any developer improving documentation on a legacy module.
- Any project transitioning from exploration to stabilization.
- Any developer changing role and becoming more pedagogical in their code.

### False Negatives
- Copilot users who systematically remove generated comments.

### Trivial Bypass
Delete all comments before committing.

---

## 3. Detection by Commit Messages

### Description
Commit messages formatted as Conventional Commits
(`feat:`, `fix:`, `chore:`) are supposedly associated with LLMs.

### Bias
- Conventional Commits is a documented convention adopted by thousands of projects
  well before the LLM era.
- Tools like `commitizen`, `semantic-release`, and GitHub templates enforce this
  format independently of AI.
- Commit message style is often mandated by reviewers or CI.

### False Positives
- Any project using `semantic-release` or `commitizen`.
- Any developer who reads Git best practices (post-2020).
- Any open-source project that requires structured messages in CONTRIBUTING.md.

### False Negatives
- Copilot users who write commit messages manually (which many do).

### Trivial Bypass
Write commit messages manually.

---

## 4. Detection by Naming Conventions

### Description
LLMs generate longer, more descriptive variable and function names
(`calculate_user_total_purchase_amount` vs `calc_total`).

### Bias
- Naming conventions vary by language, framework, and team guidelines.
- Python, for example, has always favored descriptive names (PEP8).
- A developer improving readability on a legacy codebase produces exactly this signal.
- Automatic renaming tools (e.g., IDE refactoring) produce the same signal.

### False Positives
- Any developer adopting modern language conventions.
- Any senior developer reviewing junior code.

### False Negatives
- An LLM user who prefers short names by habit and imposes them on suggestions.

### Trivial Bypass
Rename generated variables to match personal style before committing.

---

## 5. Detection by Formatting

### Description
Detecting specific formatting patterns (spacing, indentation, line length)
associated with LLMs.

### Bias
- Automatic formatters (black, prettier, rustfmt) make code indistinguishable
  from LLM-formatted code.
- Adopting a formatter in an existing project produces exactly this signal.

### False Positives
- Any project that adopts an auto-formatter.
- Any developer who reconfigures their editor.

### False Negatives
- Any LLM user whose formatter reformats the suggestions.

### Trivial Bypass
Run generated code through a formatter before committing.

---

## 6. Searching for ChatGPT Boilerplate

### Description
Searching for phrases like "Certainly!", "As an AI language model", "Sure, here's",
or docstring patterns specific to GPT-4.

### Major Bias
- These phrases appear in the conversational responses of LLMs, not in the code
  they generate.
- If someone requests only a code snippet, the LLM will not preface its response
  with "Certainly!".

### Fundamental Invalidity
This approach has a true positive rate near zero for pure code.
It is only applicable to commit messages or README files generated with a prompt
that includes prose, which is rare in real software development.

---

## 7. Single-Commit Detection

### Fundamental Invalidity
A single commit can never provide statistical evidence of anything.

**Reasons:**
1. The inter-commit variance of a single developer exceeds the difference between
   human code and AI code (noise > signal).
2. There is no reference distribution for "human code" vs. "AI code" —
   the two distributions overlap massively.
3. A single commit is susceptible to all the bias sources cited above.
4. Inference from a single commit violates the fundamental statistical principle:
   one observation cannot establish a pattern.

**What can be detected from a single commit:** nothing defensible.

**What can be detected from a time series:** a change in development behavior
relative to the author's historical baseline.

---

## Conclusion: What This Project Does Differently

This project abandons the approaches above and focuses on:

1. **Process signals (Level A)** — measuring HOW code is written, not the result:
   commit frequency, median size, cross-module dispersion, refactoring ratio.

2. **Longitudinal analysis** — comparing the author against their own past,
   not against a generic "human code" distribution.

3. **Explicit statistical tests** — Mann-Whitney U for distribution comparison,
   Fisher's method for signal combination.

4. **Probabilistic interpretation** — P(behavioral drift | history),
   with explicit statement of alternative hypotheses.

5. **Acknowledgment of limits** — without ground truth data (developers' explicit
   declarations), no calibration is possible.
